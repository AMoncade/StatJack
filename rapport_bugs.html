Critique #1 — tirer() retourne None sans gestion dans plusieurs chemins jeu.py ▶

Fichier & lignes

src/models/jeu.py — lignes 40-43, 49, 61, 78-79, 95

Description

demarrer_manche() appelle self.sabot.tirer() 4 fois sans verifier si le retour est None. Si le sabot a moins de 4 cartes mais plus de 20 (seuil de reinitialisation a la ligne 37), tirer() retourne None et ajouter_carte(None) ajoute None a la main. Ensuite, valeur_blackjack() sur None crash avec AttributeError.

Meme probleme dans joueur_tire() (ligne 49), tour_du_dealer() (ligne 95), et joueur_split() (lignes 78-79).

Declencheur

Le sabot a entre 0 et 19 cartes restantes (pas assez pour reinitialiser) et plusieurs tirages l'epuisent en cours de partie. Improbable avec 6 paquets mais possible avec 1-2 paquets et plusieurs splits.

Correction suggeree

Verifier le retour de chaque tirer(), ou augmenter le seuil de reinitialisation (ex: < 30) pour couvrir le pire cas.

Majeur #3 — Attribut de classe rang masque par l'attribut d'instance carte.py ▶

Fichier & lignes

src/models/carte.py — lignes 6, 9

Description

La classe a rang = ["2", "3", ...] comme attribut de classe (ligne 6) et self.rang = rang dans __init__ (ligne 9). Apres instanciation, instance.rang retourne la string (ex: "K"), pas la liste. Si du code itere sur some_carte.rang en attendant la liste de classe, il itere sur les caracteres de la string.

Declencheur

Tout code qui accede a .rang sur une instance en attendant la liste.

Correction suggeree

Renommer l'attribut de classe en RANGS pour eviter le masquage.

Majeur #4 — retirer_carte() ne met pas a jour le running_count sabot.py ▶

Fichier & lignes

src/models/sabot.py — lignes 56-64

Description

Quand retirer_carte() retire une carte du sabot, le running_count n'est pas mis a jour avec la valeur Hi-Lo de cette carte. Le comptage devient donc inexact apres tout retrait via cette methode.

Declencheur

Appele par retirer_cartes() (ligne 66-69), pas utilise dans le flux principal mais pourrait l'etre dans les calculs de probabilite ou futures fonctionnalites.

Correction suggeree

Ajouter self.running_count += carte.valeur_hilo() apres le pop.

Majeur #6 — Race condition : threads de workers orphelins sur clics rapides worker_probas.py / controleur_jeu.py ▶

Fichier & lignes

src/controllers/workers/worker_probas.py — lignes 17-19
src/controllers/controleur_jeu.py — ligne 243

Description

Quand le joueur clique rapidement (hit, hit, hit), un nouveau worker est cree a chaque fois. L'ancien worker continue de tourner en arriere-plan. Le _id_calcul_probas protege contre les resultats obsoletes, mais les threads orphelins font du Monte Carlo lourd, causant du lag UI et de la pression memoire.

Declencheur

Clics rapides sur Hit/Stand pendant le jeu.

Correction suggeree

Verifier si l'ancien worker tourne encore et le marquer pour arret anticipe (flag _cancelled dans les boucles de simulation).

Majeur #7 — Mode entrainement bloque le UI avec calcul synchrone controleur_jeu.py ▶

Fichier & lignes

src/controllers/controleur_jeu.py — lignes 304-332

Description

_verifier_decision_mathematique() appelle CalculateurProbabilites.resume_spot() de maniere synchrone sur le thread principal avec 1000 simulations Monte Carlo. L'interface gele pendant plusieurs centaines de millisecondes a plusieurs secondes.

Declencheur

Activer le mode entrainement, puis cliquer Hit ou Stand.

Correction suggeree

Executer le calcul dans un worker thread, ou reutiliser la recommandation du worker asynchrone deja en cours.

Majeur #8 — mode_entrainement absent de DEFAUTS, perdu au reset settings.py / controleur_jeu.py ▶

Fichier & lignes

Description

La cle mode_entrainement n'est pas declaree dans Settings.DEFAUTS. settings.get("mode_entrainement") retourne None. Le reglage ne survit pas a un settings.reset() : la cle est completement effacee mais la checkbox UI garde l'ancien etat.

Declencheur

Activer le mode entrainement, puis reinitialiser les parametres.

Correction suggeree

Ajouter "mode_entrainement": False a DEFAUTS.

Majeur #9 — Changer nb_paquets ne met pas a jour le sabot en cours vue_parametres.py / main.py ▶

Fichier & lignes

src/views/vue_parametres.py — lignes 70-72
main.py — lignes 75-77

Description

Quand l'utilisateur change nb_paquets dans le spinner, c'est ecrit dans les settings mais le jeu.sabot n'est PAS mis a jour. Le sabot n'est recree que quand aller_jouer() est appele. Si le changement est fait depuis l'ecran de jeu, il n'a aucun effet jusqu'au retour au menu.

Declencheur

Changer le nombre de paquets depuis les parametres pendant une partie.

Correction suggeree

Desactiver le changement pendant une partie, ou recreer le sabot au retour des parametres.

Majeur #11 — Double/Split ignorent la verification du mode entrainement controleur_jeu.py ▶

Fichier & lignes

src/controllers/controleur_jeu.py — lignes 81-119

Description

action_double() ne verifie PAS mode_entrainement comme action_hit() et action_stand() le font. Un joueur en mode entrainement peut doubler et splitter sans etre valide contre la strategie optimale.

Declencheur

Activer le mode entrainement, puis cliquer Double ou Split.

Correction suggeree

Ajouter la meme verification _verifier_decision_mathematique dans action_double() et action_split().

Majeur #12 — La recommandation ne suggere jamais DOUBLE ou SPLIT probabilites.py ▶

Fichier & lignes

src/models/probabilites.py — ligne 260

Description

resume_spot() retourne "HIT" ou "STAND" uniquement. La strategie optimale devrait aussi considerer DOUBLE et SPLIT. Le code ne recommande jamais ces actions meme quand elles sont mathematiquement correctes (ex: 11 vs dealer 6, paire de 8).

Declencheur

Toute main ou doubler ou splitter est le jeu optimal.

Correction suggeree

Calculer l'EV pour double (une carte, paiement double) et split, puis comparer toutes les options.

Mineur #2 — Troncature int() sur le paiement Blackjack pour mises impaires jeu.py ▶

Fichier & lignes

src/models/jeu.py — ligne 116

Description

gain = main.mise + int(main.mise * 1.5) utilise int() qui tronque. Pour mise=101: 101 + int(151.5) = 252 au lieu de 252.5. Non declenche actuellement car les jetons sont en multiples de 100.

Correction suggeree

Utiliser main.mise * 5 // 2 ou round().

Mineur #14 — Les egalites comptees comme pertes dans le Monte Carlo probabilites.py ▶

Fichier & lignes

src/models/probabilites.py — lignes 370-374

Description

La simulation Monte Carlo ne compte que les victoires strictes. Les egalites (push) sont comptees comme des non-victoires, donc le "% de victoire" affiche est sous-estime. Le label dit "(Gagner)" donc c'est techniquement correct mais potentiellement trompeur.

Correction suggeree

Reporter victoire/egalite/defaite separement, ou clarifier dans l'UI.

Mineur #15 — Fuite de descripteur fichier si exception dans setSource() audio.py ▶

Fichier & lignes

src/models/audio.py — lignes 30-36

Description

Le bloc de suppression stderr (os.open, os.dup, os.dup2, os.close) n'est pas dans un try/finally. Si setSource() lance une exception, les descripteurs ne seront pas fermes et stderr restera redirige.

Correction suggeree

Envelopper dans try/finally pour toujours restaurer stderr.
